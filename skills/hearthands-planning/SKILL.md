# Hearthands Planning

Use this when work needs to be broken into durable steps before implementation, delegation, or handoff.

## Principle

Clear handoff is care. A plan should reduce uncertainty for the next worker without pretending the future is more knowable than it is.

## Pattern

1. Restate the goal in one sentence.
2. Separate known requirements from assumptions.
3. Identify the files, docs, commands, or systems that anchor the work.
4. Split the work into steps that each produce a reviewable change.
5. Put verification next to the step it proves.
6. Name dependencies, risks, and stop points.
7. Keep the plan small enough to execute and specific enough to review.

## Task Shape

Each task should answer:

- What changes?
- Where does it change?
- What evidence proves the task worked?
- What should not be touched?
- What context would prevent a future worker from getting stranded?

## Guardrails

- Do not use planning to inflate scope.
- Do not hide uncertainty behind rigid sequencing.
- Do not split work so finely that review loses meaning.
- Do not combine unrelated risks into one task.
- Do not write placeholders where a future worker needs judgment, commands, or concrete paths.

## Output

Produce a plan with goal, assumptions, task list, file targets, verification checks, risks, and handoff notes. If the work is not ready to plan, name the missing information and the smallest next discovery step.
